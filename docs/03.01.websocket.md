# WebSocket 功能與用法介紹

本專案支援透過 WebSocket 作為通訊管道 (Channel)，讓 AI 助理 (pcai) 能夠連線至外部的中繼伺服器，接收用戶的訊息並即時回覆。

## 1. 原生 WebSocket Channel (`internal/channel/websocket.go`)

專案內建了原生的 WebSocket channel 實作，作為 AI 的其中一個輸入與輸出端點。這使得本系統可以輕易地整合到任何提供 WebSocket 介面的聊天室或前端網頁應用中。

### 1.1 設定與啟用方式

要啟用 WebSocket Channel，需透過環境變數或設定檔 (對應 `internal/config/config.go`) 來配置：

- `WEBSOCKET_ENABLED`: 設為 `true` 以啟用 (預設為 `false`)。
- `WEBSOCKET_URL`: 指定要連線的 WebSocket 伺服器網址。預設為 `wss://jarvis.justdrink.com.tw/webhook`。

**範例環境變數配置：**
```bash
WEBSOCKET_ENABLED=true
WEBSOCKET_URL=wss://your-websocket-server.com/ws
```

### 1.2 連線機制與容錯

- **自動重連：** 啟動後若連線失敗，會自動進入重試機制。初始延遲為 2 秒，失敗後會指數增加至最高 30 秒。
- **Keep-Alive (Ping/Pong)：** 連線成功後，會啟動背景常駐程序，每 30 秒向伺服器發送一次 Ping 控制訊息，確保連線活絡，避免被伺服器因閒置而斷開。
- **優雅關閉 (Graceful Shutdown)：** 當系統關閉時，會發送關閉訊號取消連線與相關 goroutine。

### 1.3 訊息通訊格式 (JSON)

用戶端或中繼伺服器與本系統通訊時，需採用特定的 JSON 格式 (`WSMessage`)：

```json
{
  "channel": "pcai",      // 必須為 "pcai"，系統僅會處理標籤符合的訊息
  "user_id": "用戶ID",    // 發送者的唯一識別碼
  "reply_to": "回覆ID",    // (選用) 用於標記回覆對象
  "message": "輸入訊息",   // 用戶發送給 AI 的純文字內容
  "type": "訊息類型"       // 若為 AI 的回覆，則 type 為 "response"
}
```

- **處理邏輯：** 系統只會處理 `channel` = `"pcai"` 並且 `type` **不等於** `"response"` 的訊息。這能有效避免 AI 接收到自己剛送出的訊息而產生無窮迴圈回覆的問題。
- **回應邏輯：** 當 AI 代理人處理完畢後，會將結果包裝成相同的 JSON 格式，並將 `type` 設為 `"response"`，寫入 WebSocket 通道回傳給伺服器。

## 2. Signal CLI WebSocket 接收端 (`socketio.go`)

在 codebase 中還存在一個特別的 WebSocket 用途——連接 Signal 伺服器：

- 這部分並不是傳統意義上作為通用 Channel 的 WebSocket，而是專門用於與 `signal-cli-rest-api` 掛鉤。
- 它連線到 `ws://<SignalHost>/v1/receive/<SignalNumber>`，即時接收來自 Signal 傳訊軟體的訊息。
- 收到訊息後，會經過 JSON 解析並抽取 `dataMessage`，然後與 AI `ChatSession` 互動，最後透過 HTTP POST 調用 `/v2/send` 介面將 AI 回覆發送回 Signal 聯絡人。

*(註：若僅需要標準的網頁 WebSocket 對接，請以第 1 節的 `WebSocketChannel` 為主。)*
